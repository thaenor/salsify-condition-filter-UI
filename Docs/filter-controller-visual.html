Derived State computed on every render, never stored
Why derived? The draft is the single source of truth. Everything below is calculated from it. No stale state, no sync bugs.
validOperators
derived
Which operators appear in the dropdown. Determined by the selected property's type, looked up in the compatibility matrix.
COMPATIBILITY[property.type] → filtered from operators catalog
Property = "Product Name" (string)
validOperators = Equals, Has any value, Has no value, Is any of, Contains
Property = "weight (oz)" (number)
validOperators = Equals, Is greater than, Is less than, Has any value, Has no value, Is any of
Property = "category" (enumerated)
validOperators = Equals, Has any value, Has no value, Is any of
No property selected
validOperators = [] — operator dropdown hidden

filteredProducts
derived
What the table renders. If the draft is ready, applies the filter. Otherwise, all products.
isReady(draft) ? applyFilter(products, toCriteria(draft)) : products
Draft is ready: "Product Name" + "Contains" + "phone"

Draft is NOT ready (any other stage)

The Reset Cascade the one rule that ties it together

Each level resets everything below it

selectProperty → operator gone, value gone, error gone
selectOperator → value gone, error gone
setValue (fail) → error set, stays at needs-value
clear → everything gone
Why? A new property might not support the old operator (e.g. switching from number to string loses "greater_than"). A new operator might change the value input kind (e.g. "equals" needs text, "is any of" needs multi-text). Resetting downstream prevents impossible combinations.
